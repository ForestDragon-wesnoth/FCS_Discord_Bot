VTT Bot Test Sequences
======================

Each scenario starts fresh with its own match id (m1, m2, ...). You can run
any scenario independently, or run all in sequence and they won't collide.
Quote any argument containing spaces, just like in a shell.


SCENARIO 1 — Setup, formulas, and inline $() substitution
----------------------------------------------------------

!match new m1 demo 8 8
!match use m1
!ent add hero Hero 30 4 4 10
!ent add rogue Rogue 25 5 5 20
!list
!eval "entity[rogue].hp + entity[hero].hp"
!eval "min(entity[hero].hp, entity[rogue].hp)"
!ent set_var hero regen 7
!ent hp hero "$(entity[this].hp * -1 + 5)"
!eval "entity[hero].hp"
!ent hp hero "$(entity[self].regen)"
!eval "entity[hero].hp"

Expected: list shows two entities; sum eval returns 55; min returns 25; the
$(entity[this].hp * -1 + 5) expression damages Hero by 25 (HP 30 -> 5);
$(entity[self].regen) then heals by 7 (HP 5 -> 12).

SCENARIO 26 — Overheal demo: full mechanic with on_var_write_attempt
---------------------------------------------------------------------

This scenario walks through the three-stage story of overheal tracking:

(A) The existing was_clamped + intended_value mechanism handles "heal
    from below into cap" correctly.
(B) That mechanism misses heals at full HP because they produce no diff.
(C) on_var_write_attempt closes the gap.

!match new m26 overheal_demo 8 8
!match use m26
!ent add hero Hero 30 4 4 10

Setup: Hero has hp=30/30 by default.

Stage A: on_var_changed catches heal-from-below clamping

!ent set_var hero overheal_changed 0
!passive add hero track_changed on_var_changed target=hp scope=exact "entity[self].overheal_changed = entity[self].overheal_changed + (intended_value - new_value) if was_clamped else entity[self].overheal_changed"
!ent set_var hero hp 5
!ent hp hero 100
!ent info hero

Expected: hp goes from 5 toward 105 but the default soft clamp engages at
max_hp=30. The on_var_changed event fires with intended_value=105 and
new_value=30, so overheal_changed = 75. This part works perfectly with
just the existing hooks.

Stage B: the gap — heal at full HP produces NO on_var_changed event

!ent hp hero 50
!ent info hero

Expected: hp stays at 30 (already at cap, soft clamp dormant from below
since old=30=cap, write 80 → clamped to 30 — but 30==30 means the diff is
empty and on_var_changed does NOT fire). overheal_changed is still 75
because the tracker never ran. So 50 wasted heal points went unrecorded.
This is the gap the user identified.

Stage C: on_var_write_attempt closes the gap

!ent set_var hero overheal_attempt 0
!passive add hero track_attempt on_var_write_attempt target=hp scope=exact "entity[self].overheal_attempt = entity[self].overheal_attempt + (intended_value - new_value) if intended_value > new_value else entity[self].overheal_attempt"
!ent set_var hero hp 5
!ent hp hero 100
!ent info hero
!ent hp hero 50
!ent info hero

Expected: overheal_attempt = 75 + 50 = 125. The attempt hook fires for
EVERY write to hp regardless of whether the diff produces a change event.
The first heal (5 → clamped to 30) contributes 75; the second heal (30 →
clamped to 30) contributes 50. The user sees the full 125 wasted points.

Why two hooks exist: on_var_changed says "the data actually changed,"
on_var_write_attempt says "a write was attempted." Most game mechanics
want one or the other:
  - HP regen tracking only cares about actual hp gains: on_var_changed
  - Damage taken: on_var_changed (only when hp actually goes down)
  - Overheal accounting: on_var_write_attempt (count all wasted heal)
  - "Audit every interaction with this var": on_var_write_attempt
The on_var_written catch-all subscribes to created/changed/removed but
NOT to attempts, so subscribe to both write_attempt AND written if you
want truly comprehensive auditing.

SCENARIO 66 — Mana shield: damage drains mana first, hp only when mana=0
-------------------------------------------------------------------------

!match new shield66 ManaShield 6 6
!match use shield66
!ent add hero Hero 30 3 3 10
!ent set_var hero mana 20
!ent set_var hero max_mana 50
!passive add hero mana_shield on_var_changed target=hp scope=exact "if new_value < old_value:\n    entity[self].mana = max(0, entity[self].mana - (old_value - new_value))\n    entity[self].hp = old_value - max(0, (old_value - new_value) - old_mana_amount) if entity[self].mana > 0 else new_value"

This is getting complex. A simpler version:

!passive remove hero mana_shield
!passive add hero mana_absorbs on_var_changed target=hp scope=exact "if new_value < old_value and entity[self].mana > 0:\n    entity[self].hp = old_value\n    entity[self].mana = max(0, entity[self].mana - (old_value - new_value))"

!ent hp hero -15
!ent info hero
!ent hp hero -15
!ent info hero

Expected: When hp drops, mana absorbs the damage (hp is restored, mana

to 5. Next 15 damage: hp would drop to 15, but mana>0 absorbs it (mana
to 0, hp restored). Once mana=0 the passive's condition fails and
damage applies normally.

Note: the mana_absorbs passive itself writes hp, which retriggers
on_var_changed. To avoid infinite loops, the condition mana > 0 must
become false before recursion reaches the limit.

SCENARIO 175 — random_string with if/elif/else branching
---------------------------------------------------------

!match new mrand2 RandStr 8 8
!match use mrand2
!ent add hero Hero 100 1 1 10
!ent set_var hero status none
!ent set_var hero buff 0
!passive add hero mood on_turn_start "entity[self].status = random_string('rage', 'calm', 'fear'); entity[self].hp = max(0, entity[self].hp - 8) if entity[self].status == 'rage' else (min(entity[self].max_hp, entity[self].hp + 4) if entity[self].status == 'calm' else entity[self].hp); entity[self].buff = 0 if entity[self].status == 'rage' else (5 if entity[self].status == 'calm' else -2)"
!turn next
!ent info hero
!turn next
!ent info hero
!turn next
!ent info hero
!turn next
!ent info hero

Expected: The `mood` passive rolls a status on each turn-start —
one of 'rage', 'calm', 'fear' — stores it on the entity, then
branches:
  - rage   →  HP -8, buff cleared to 0
  - calm   →  HP +4 (clamped to max_hp), buff = 5
  - fear   →  buff = -2 (HP unchanged; the entity is too scared
              to swing but takes no damage either)

The formula above uses chained ternaries to fit on one line — when
typed into Discord directly the same logic reads more naturally as
if/elif/else:

    entity[self].status = random_string("rage", "calm", "fear")
    if entity[self].status == "rage":
        entity[self].hp = max(0, entity[self].hp - 8)
        entity[self].buff = 0
    elif entity[self].status == "calm":
        entity[self].hp = min(entity[self].max_hp, entity[self].hp + 4)
        entity[self].buff = 5
    else:
        entity[self].buff = -2

Both forms have identical semantics. Across four turns, the hero's
`status` and `buff` vars track whichever branch fired most recently
and HP drifts down or up depending on which moods came up. Because
random_string is non-deterministic the exact sequence varies, but
every turn the status is one of the three listed values and the
HP/buff pair is consistent with the branch that name implies — a
useful pattern for "random table" status effects.

The roll is assigned to entity[self].status FIRST, then the
ternary chain (or the if/elif) reads it back. Calling random_string
twice in a row would give two independent rolls, which is rarely
what a GM wants for a "roll once and react" pattern — using an
entity var (or any formula-readable storage) is the idiom for
memoizing the roll.

command which validates the full destination in one shot. The
last !ent info confirms A is still at (1,1) — the rejected
writes didn't change anything because the engine never started
the move.

SCENARIO 180 — Formula access to tile data via tile_get / tile_has
-------------------------------------------------------------------

!match new mtile2 TileFormula 8 8
!match use mtile2
!ent add hero Hero 30 1 1 10
!tile set 3 3 flame.burn_damage 5
!tile set 3 3 flame.spreads 1
!tile set 5 5 spikes.damage 8
!eval 'tile_get(3, 3, "flame.burn_damage")'
!eval 'tile_get(3, 3, "flame.spreads")'
!eval 'tile_has(3, 3, "flame.burn_damage")'
!eval 'tile_has(3, 3, "spikes.damage")'
!eval 'tile_has(99, 99, "flame.burn_damage")'
!eval 'tile_get(3, 3, "spikes.damage")'
!ent tp hero 3 3
!eval 'entity[hero].hp - tile_get(entity[hero].x, entity[hero].y, "flame.burn_damage")'
!passive add hero burn on_turn_start "entity[self].hp = entity[self].hp - tile_get(entity[self].x, entity[self].y, \"flame.burn_damage\") if tile_has(entity[self].x, entity[self].y, \"flame.burn_damage\") else entity[self].hp"
!turn next
!ent info hero
!ent tp hero 1 1
!turn next
!ent info hero
!ent tp hero 5 5
!turn next
!ent info hero

Expected: tile_get returns the value at a dotted path on a tile;
missing paths raise ❌ FormulaError ("tile (3,3) has no value at
'spikes.damage'") so the GM gets a precise error rather than the
formula silently returning None. tile_has is the boolean guard —
returns True/False, never raises, and returns False for both
"path missing" and "coordinate out of grid" so a formula can
defensively pass entity-derived coords without first bounds-checking
them.

The chained read `tile_get(entity[hero].x, entity[hero].y,
"flame.burn_damage")` is the user-facing pattern: "what's the burn
damage on whatever tile this entity is standing on?". With Hero
teleported to (3,3), the eval returns 30 - 5 = 25 (hero.hp minus
the tile's burn_damage).

The `burn` passive bakes that into an on_turn_start hook with
tile_has as a guard so the passive is a no-op when the entity
isn't on a fire tile. Across three turn-nexts the hero's HP path
is:
  - on (3,3): turn fires, has flame → -5, hp = 25
  - on (1,1): turn fires, no tile → no damage, hp = 25
  - on (5,5): turn fires, has spikes but NOT flame.burn_damage →
              guard is False, no damage, hp = 25
Even though (5,5) has spikes.damage, the passive only fires when
flame.burn_damage exists — exact path matching, no wildcards. A
more complete fire-or-spikes passive would chain two tile_has
checks via if/elif.

Note the shell quoting: the formula's string literal (the dotted
path argument) needs inner double quotes to survive shlex
tokenization. Wrap the whole !eval argument in outer single
quotes so the inner double quotes reach the formula parser
intact — without the outer single quotes, shlex strips the inner
ones and the formula sees `flame.burn_damage` as an attribute
access on an undefined `flame` identifier, which the validator
rejects.

Adding Carol mid-game inserts her at the right initiative

-----------------------------------------------------------------------

!match use mfn4
!ent add hero Hero 30 1 1 10
!ent set_var hero max_hp 50
!func def regen_amount missing "ceil(missing / 4)"
!passive add hero regen on_turn_start "entity[self].hp = min(entity[self].hp + regen_amount(entity[self].max_hp - entity[self].hp), entity[self].max_hp)"
!ent hp hero -20
!ent dump hero
!turn next
!ent dump hero

Expected: A passive body can call a formula function, and inside
that function `self` correctly resolves to the passive's owner.
`regen_amount(missing)` returns `ceil(missing / 4)`. Hero starts
at 30/50, takes 20 damage to 10/50 (missing = 40). When the
on_turn_start passive fires, it computes
`regen_amount(40)` = `ceil(40/4)` = 10 and heals to 20/50.

This is the headline use case: define reusable math once
(`regen_amount`), reference it from any number of passives, and
edit the formula in one place. The function sees the calling
passive's `self`, so it works for any entity the passive is
attached to.

SCENARIO 222 — Formula functions: persistence and deletion
-----------------------------------------------------------

!match new mfn6 FuncPersist 8 8
!match use mfn6
!func def triple x "x * 3"
!func def viafn x "triple(x) + 1"
!eval "viafn(10)"
!store save /tmp/scenario222
!func del triple
!eval "viafn(10)"
!func list
!store load /tmp/scenario222
!func list
!eval "viafn(10)"

Expected: Formula functions survive a save/load round-trip.
`triple(x)` = `x * 3` and `viafn(x)` = `triple(x) + 1`;
`viafn(10)` = 31. After saving, deleting `triple` breaks `viafn`
— calling it now errors because `triple` no longer resolves (no
dependency scan is done at delete time; dependents fail at their
next call). `!func list` shows only `viafn` remains. Loading the
saved state restores both functions, and `viafn(10)` = 31 works
again. (The deletion's downstream breakage is intentional and
mirrors how tile-template deletion leaves orphaned instances.)


SCENARIO 223 — Formula functions: built-in protection and overwrite reporting
------------------------------------------------------------------------------

!match new mfn7 FuncGuard 8 8
!match use mfn7
!func def max x "x + 1"
!func def clamp x "x"
!func def distance a "a"
!func def is_hostile a "a"
!func def group_add a "a"
!func def tile_get a "a"
!func def mitigated raw,armor "max(raw - armor, 0)"
!func def mitigated raw,armor "max(raw - armor * 2, 0)"

Expected: Built-in formula functions are protected from

accidental overwrite is visible. The second and third

signature+body ("was:") and the replacement ("now:"). The
parameter list can change across a redefinition (the third call
adds a `pierce` parameter). The final `mitigated(20, 5, 3)`
confirms the latest definition is the active one:
max(20 - 5 + 3, 0) = 18.

!match new mdyn2 DynParam 12 12

Expected: Inside a function body, a parameter used as an entity index

SCENARIO 234 — Seedable RNG: reproducible roll sequences
---------------------------------------------------------

!system set default random_seed encounter42
!match new mseed1 Seeded1 8 8
!match use mseed1
!eval "random_int(1, 1000)"
!eval "random_int(1, 1000)"
!eval "random_int(1, 1000)"
!match new mseed2 Seeded2 8 8
!match use mseed2
!eval "random_int(1, 1000)"
!eval "random_int(1, 1000)"
!eval "random_int(1, 1000)"
!system set default random_seed

Expected: With the `random_seed` rule set to a non-empty value, the
match's formula RNG becomes deterministic — each match seeded with
the same value produces the SAME sequence of rolls. So the three
rolls in Seeded1 exactly match the three rolls in Seeded2
(same seed "encounter42", same call order). The RNG advances across
calls (the three rolls within a match differ from each other — it's
a reproducible SEQUENCE, not a stuck value). The trailing
`!system set default random_seed` with no value clears the seed
back to unseeded (independent rolls) so later scenarios aren't
affected. (Test note: because the two matches share the same seed
and the same number of prior rolls, their roll lists are equal —
that equality is the deterministic assertion; the exact numbers
depend on the seed.)

SCENARIO 230 — !tile place_random: scatter instances across empty cells

Expected: `!tile place_random <template> <count>` scatters `count`

(entity-occupied cells are still eligible — tiles and entities

SCENARIO 231 — !tile fill_pattern: paint an ASCII stencil
----------------------------------------------------------

!match new mtr2 FillPattern 8 8
!match use mtr2
!tile def new wall
!tile def data wall glyph #
!tile def new water
!tile def data water glyph ~
!tile fill_pattern 2 2 "#=wall,~=water" "##~" "#.~" "~~~"
!tile find wall
!tile find water
!tile info 3 3
!tile fill_pattern 7 7 "#=wall" "###" "###"
!tile fill_pattern 1 1 "#=nope" "#"
!tile fill_pattern 1 1 "##=wall" "#"

Expected: `!tile fill_pattern <x> <y> <legend> <rows...>` paints a
multi-row stencil with its top-left at (x, y). The legend maps
single characters to templates; each row places one template per
character at (x + column, y + row). Characters not in the legend
are skipped (left untouched) — '.' is the conventional hole.

The 3x3 stencil at (2,2):
  row "##~" → (2,2) wall, (3,2) wall, (4,2) water
  row "#.~" → (2,3) wall, (3,3) SKIP,  (4,3) water
  row "~~~" → (2,4) water,(3,4) water, (4,4) water
So `!tile find wall` shows 3 cells, `!tile find water` shows 5,
and `!tile info 3 3` reports "no data" (the '.' left it empty).

The remaining calls demonstrate all-or-nothing validation: a
stencil at (7,7) that would spill off the 8x8 grid fails with an
out-of-bounds cell error (placing nothing); a legend referencing
an undefined template (`nope`) fails up front; a multi-character
legend key (`##`) is rejected.
